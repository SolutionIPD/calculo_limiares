1. Visão Geral e Metodologia

Introdução

Este dashboard descreve o processo de cálculo dos limiares de chuva acumulada (em mm) utilizados para a emissão de alertas de risco de deslizamento. O objetivo é consolidar o conhecimento sobre a metodologia, os dados envolvidos e os resultados gerados, servindo como base para a evolução do sistema.

Mapeamento de Dados

O processo depende das seguintes fontes:

  • Parâmetros Pré-calculados: Arquivo tb_params_automatico.rds contendo os parâmetros (mu, phi, power) da distribuição Tweedie para diversas estações.
  • Dados de Novas Estações: Estações sem parâmetros pré-definidos (Veranópolis, Barão, etc.) utilizam os parâmetros da estação "BENTO GONCALVES" como aproximação.
  • Dados Históricos: Coletados de fontes como o INMET. A qualidade destes dados é fundamental para a precisão do modelo.

Metodologia Estatística: Distribuição Tweedie

O pilar do cálculo é a distribuição Tweedie, ideal para modelar dados de precipitação, pois lida com valores zero (dias sem chuva) e contínuos positivos (dias com chuva).

O script consome os parâmetros pré-calculados e utiliza a função quantílica (tweedie::qtweedie) para encontrar os valores de chuva que correspondem aos seguintes percentis (cuts):

  • 65%: Limiar “Moderado”
  • 85%: Limiar “Alto”
  • 95%: Limiar “Muito Alto”
  • 99%: Limiar “Altíssimo”

# Combina os parâmetros originais com os das novas estações.
tb_params <- bind_rows(tb_params_novas_estacoes, tb_params_originais) |>
 distinct(station, horas, .keep_all = TRUE)

# Função que calcula os limiares de chuva (em mm)
susceptibility_limits <- function(tb_params, station, hours = 96, cuts) {
 params <-
   tb_params |>
   filter(station == !!station, horas == !!hours) |>
   select(c("mu", "phi", "power")) |>
   as.list()

 if (length(params$mu) == 0) return(rep(NA, length(cuts)))

 bks <- tweedie::qtweedie(cuts, mu = params$mu, phi = params$phi, power = params$power)
 return(bks)
}

# --- Cálculo e Exibição dos Limiares ---
limiares_por_estacao <-
 unique(tb_params$station) |>
 map_df(~ {
   limites <- susceptibility_limits(tb_params, station = .x, hours = 96, cuts = cuts)
   tibble(
     Estacao = .x,
     `Moderado (mm)` = limites[1],
     `Alto (mm)` = limites[2],
     `Muito Alto (mm)` = limites[3],
     `Altíssimo (mm)` = limites[4]

3. Geração de Parâmetros e Comparativo

# Definição da Função Corrigida
fit_tweedie_corrigido <- function(y) {
   if (all(y == 0) || length(unique(y)) < 2) {
       return(list("mu" = 0, "phi" = 1, "power" = 1.5))
   }
   data_df <- data.frame(y = y)
   power_profile <- try({
       capture.output(
           obj <- tweedie::tweedie_profile(y ~ 1, data = data_df, p.vec = seq(1.1, 1.9, by = 0.1), do.plot = FALSE, verbose = FALSE),
           type = c("output", "message")
       )
       obj
   }, silent = TRUE)
   p_otimo <- if (inherits(power_profile, "try-error")) 1.5 else power_profile$p.max
   modelo_glm <- glm(y ~ 1, data = data_df, family = tweedie(var.power = p_otimo, link.power = 0))
   params <- list(
      power = p_otimo,
      mu = exp(coef(modelo_glm)),
      phi = summary(modelo_glm)$dispersion
    )
   return(params)
}

  2. Método 2 (RDS Bento Gonçalves): Parâmetros originais da estação “BENTO GONCALVES”, usados como “fonte da verdade” para a simulação e comparação.

4. Planejamento e Próximos Passos

A análise do processo atual levanta questões importantes para a evolução do sistema de alertas.

  • Separação de Processos (Treinamento vs. Cálculo): É recomendável desacoplar o treinamento do modelo (geração de parâmetros, que pode rodar com baixa frequência) do cálculo de limiares (que consome os parâmetros e roda conforme a necessidade). Isso traria mais robustez e manutenibilidade?

  • Automatização para Novas Estações: A prática de “emprestar” parâmetros é uma solução paliativa. Um fluxo automatizado para calcular parâmetros para novas estações, assim que tiverem dados suficientes, aumentaria a precisão dos alertas locais. É viável e benéfico?

  • Monitoramento da Qualidade dos Dados: A precisão do modelo depende da qualidade dos dados de entrada. Falhas de coleta (SSL_ERROR_RX_RECORD_TOO_LONG) são um ponto de atenção que precisa ser tratado. Como garantir a estabilidade e qualidade da coleta de dados?
